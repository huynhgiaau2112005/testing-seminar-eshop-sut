SQLite vs MongoDB

NoSQL thay đổi công cụ, không thay đổi mục tiêu kiểm thử

SQLite

Relational Testing

  • Table và SQL schema
  • Foreign key và constraint
  • Flyway migration bằng SQL
  • V1/V2 compensating migration
  • SQL invariant tests
MongoDB

NoSQL Testing

  • Collection và JSON Schema Validator
  • Unique index và query kiểm tra
  • migrate-mongo bằng JavaScript
  • up() và down()
  • Query/aggregation invariants
Nhận xét: Schema-less không có nghĩa là không cần kiểm soát schema. MongoDB linh hoạt hơn, nhưng validation, tính toàn vẹn quan hệ và rollback phải được thiết kế rõ ràng hơn.
Lời trình bày: Khi chuyển từ SQLite sang MongoDB, mục tiêu kiểm thử của nhóm không thay đổi. Nhóm vẫn kiểm tra cấu trúc dữ liệu, dữ liệu không hợp lệ, migration, masking và chức năng sau masking. Điểm khác biệt là công cụ và cách triển khai. SQLite sử dụng Flyway và migration SQL; MongoDB sử dụng migrate-mongo với các file JavaScript có hàm up và down. MongoDB linh hoạt về cấu trúc document, nhưng schema-less không có nghĩa là không cần schema. Nhóm vẫn phải tạo JSON Schema Validator và unique index. Ngoài ra, MongoDB không tự bảo vệ foreign key như database quan hệ, nên các quan hệ giữa document phải được kiểm tra bằng query hoặc application test. Kết luận: NoSQL giảm sự cứng nhắc của schema nhưng chuyển thêm trách nhiệm kiểm soát dữ liệu sang migration code và test.

Flyway · SQL invariants · AI masking · PII · CI/CD

T08 Seminar · Database Testing
Nếu lớp im lặng: Gợi câu hỏi: “Nếu migration lỗi giữa chừng trên CI/CD thì sao?” Trả lời: ưu tiên DB hỗ trợ transactional DDL; với SQLite, tạo bản sao/backup có kiểm soát trước migration và kiểm thử khôi phục.
